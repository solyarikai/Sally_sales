# Скилл: /sales-update-project-docs

## Цель

Обновить живую базу знаний проекта в `.claude/projects/<ProjectName>/` на основе того, что произошло в текущей сессии.

---

## Шаг 0 — Определи проект

1. Определи проект по cwd или контексту разговора
2. Найди папку проекта: `.claude/projects/<ProjectName>/`
3. Если папки нет — сообщи пользователю и предложи создать

Текущие проекты:
- **OnSocial** → `.claude/projects/OnSocial/`

---

## Шаг 1 — Анализ сессии

**Не задавай вопросов.** Восстанови из переписки:

1. Что делалось в этой сессии (research, кампании, скрипты, решения)
2. Какие **новые знания** получены (инсайты о рынке, результаты тестов, данные)
3. Какие **решения** приняты (изменения стратегии, выбор инструментов)
4. Какие **автоматизации** созданы или изменены (скрипты, пайплайны)
5. Какие **кампании** запущены, остановлены, изменены

Определи какие файлы из базы знаний затронуты:

| Файл | Когда обновлять |
|------|----------------|
| `README.md` | Изменились цели, фаза, ключевые метрики |
| `market.md` | Новые данные о рынке, ICP, конкурентах, гео |
| `sequences.md` | Изменения в секвенциях, результаты A/B, новые паттерны |
| `automations.md` | Новые скрипты, изменения в пайплайне, инструменты |
| `campaigns.md` | Статус кампаний, новые запуски, метрики |
| `decisions.md` | Принятые стратегические решения с обоснованием |

---

## Шаг 2 — Прочитай текущее состояние (параллельно)

Прочитай только те файлы, которые нужно обновить (определено в шаге 1).
Не читай файлы, которые не затронуты сессией.

---

## Шаг 3 — Обнови файлы

Для каждого затронутого файла:

1. **Edit** (не Write) — точечные правки, не перезапись
2. Обнови `last_updated` в frontmatter → сегодняшняя дата
3. Добавляй новую информацию, не удаляй старую (если она не устарела)
4. Формат: факты + краткое "почему" для решений
5. Размер файла: 3-7K оптимум. Если >10K — сократи менее актуальное

### Правила по типу файла

**campaigns.md:**
- Обновляй метрики только если есть свежие данные
- Статусы кампаний: Active → Paused → Dead
- Новые кампании добавляй в соответствующий tier

**decisions.md:**
- Каждое решение: **Что** → **Why** → **Impact**
- Группируй по категориям (Segmentation, Architecture, Channel, Messaging, Operations)
- Не удаляй старые решения — они дают контекст

**sequences.md:**
- A/B результаты — обновляй при получении данных
- Новые sequences — добавляй с описанием структуры
- "What works / What doesn't" — пополняй

**automations.md:**
- Новые скрипты — добавляй в соответствующую секцию
- Изменения в пайплайне — обновляй диаграмму
- Operational notes — обновляй capacity, bugs

---

## Шаг 4 — Финальный отчёт

Выведи краткий итог (≤ 6 строк):

```
Project docs updated: <ProjectName>

Updated: <список файлов>
- <файл>: <что именно изменилось, 1 строка>

Not changed: <список файлов которые не затронуты>
```

---

## Правила

- **Не спрашивай разрешения** — анализируй и обновляй
- **Точечные правки** — Edit tool, не Write (для существующих файлов)
- **Параллельность** — читай файлы одновременно
- **Только реальные изменения** — не делай пустые правки
- **Консерватизм** — не меняй данные если не уверен в новых
- **Frontmatter** — всегда обновляй last_updated при изменении файла
- **Размер** — держи файлы в диапазоне 3-7K, максимум 10K
