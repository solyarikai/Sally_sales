## КРИТИЧЕСКИЕ ОТЛИЧИЯ (вызывают баны у нас)

### 1. FINGERPRINT MISMATCH — Android params + Desktop api_id

**Наша проблема**: Warmup worker подключается с Android-fingerprint (`Samsung SM-G998B / SDK 33 / 10.6.2`), а sending worker — с Desktop-fingerprint (`PC 64bit / Windows 10 / 6.5.1 x64`). Оба используют `api_id=2040` (TDesktop).

**Что видит Telegram**: Один и тот же аккаунт за 14 дней прогрева подключается как Android-устройство, а затем внезапно переключается на Desktop. Это явный red flag — аккаунт был передан другому пользователю/софту.

**TeleRaptor**: Использует КОНСИСТЕНТНЫЕ Desktop-fingerprints ВЕЗДЕ. Одинаковые параметры для переавторизации и для спама. Никогда не использует Android-fingerprints с Desktop api_id.

**Файлы для исправления**:
- `backend/app/services/warmup_worker.py:233-237` — заменить Android fallback на Desktop
- `backend/app/schemas/telegram_outreach.py:87-91` — заменить дефолты с Android на Desktop
- `backend/app/models/telegram_outreach.py:135-139` — заменить дефолты с Android на Desktop

### 2. ОТСУТСТВИЕ REVOKE SESSIONS

**Наша проблема**: Мы никогда не закрываем старые сессии аккаунтов. При импорте аккаунта у него могут быть 5-10 активных сессий от предыдущих владельцев/софтов.

**TeleRaptor**: Имеет отдельный модуль "Переавторизация" (`Файлы/Переавторизация/`):
- `закрывать_старые_сессии = да` (по умолчанию включено)
- `продолжать_если_не_можем_закрыть_сторонние_сессии = нет` (если не может закрыть — останавливается)
- Удаляет диалог с Telegram после переавторизации
- Устанавливает новый 2FA пароль
- Применяет новые device params

**Что видит Telegram**: У наших аккаунтов одновременно активны сессии от разных устройств, разных IP, разных городов. Это аномалия для реального пользователя.

### 3. НЕСООТВЕТСТВИЕ LANG_CODE

**Наша проблема**: Дефолтный `lang_code = "en"`, `system_lang_code = "en-US"`. Аккаунты с русскими номерами (+79...) отправляют сообщения на русском, но клиент заявляет что он англоязычный.

**TeleRaptor**: Использует `lang_code = "ru"`, `system_lang_code = "ru"`. Соответствует реальному использованию.

### B. Управление сессиями

**TeleRaptor**:
- Сессии хранятся в файлах (по телефону): `Аккаунты/Для работы/+79XXXXXXXXX.session`
- Поддерживает tdata → session конвертацию (отдельный модуль)
- Поддерживает session → tdata конвертацию
- При конвертации session → tdata: опция "не создавать новую сессию (опасно)" = нет (по умолчанию создает новую сессию!)
- Паковка: до 60 аккаунтов в одну tdata

**Наша реализация**:
- Сессии в `/app/tg_sessions/` + StringSession в БД
- Конвертация tdata → session при импорте
- Нет механизма пересоздания сессий

**Ключевое отличие**: TeleRaptor при конвертации создает НОВУЮ сессию, а не переиспользует старую. Это значит каждый аккаунт получает чистый session, привязанный к корректным device params.

**Наша реализация** (14-дневный warm-up):

**Ключевое отличие**: TeleRaptor НЕ ДЕЛАЕТ многодневный warm-up! Они просто:

Это подтверждает что наш 14-дневный warm-up НЕ ПОМОГАЕТ, потому что проблема в fingerprint mismatch: 14 дней прогрева с Android fingerprint, потом переключение на Desktop fingerprint = бан.
